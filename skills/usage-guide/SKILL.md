# Session Semantic Search

## 核心价值

过往对话的语义检索。对话的价值不在标题里，而在对话过程之中。用户可能记不清当时用了什么词、什么表述，但能描述"大概是什么事"——这正是语义搜索擅长的。

## 何时使用

当用户提出以下类型的问题时，优先使用 `session-semantic-search_search`（命名空间前缀自动添加）的 `search` 动作：

- "之前我们讨论过 XXX 相关的内容"
- "我记得你说过关于 XXX 的事，帮我找找"
- "我们是不是聊过 XXX 话题"
- "上次那个关于 XXX 的方案/想法/代码"
- "你之前分析过 XXX 的问题对吧"
- 任何涉及查阅过往对话的需求

不要等用户明确说出"搜一下历史"——只要语义上暗示需要翻看历史，就是触发电机。

## 使用流程

### 1. 检查服务状态
```
search action=health
```
确认嵌入服务（Ollama）在线。如果不在线，提醒用户启动 Ollama。

### 2. 检查索引状态
```
search action=stats
```
看索引中有多少会话和片段。如果为空（totalChunks=0），先做一次全量索引。

### 3. 首次索引（仅在需要时）
```
search action=reindex
```
全量扫描所有 session .jsonl 文件并向量化。根据会话量可能需要几分钟。

### 4. 语义搜索（核心功能）
```
search action=search query="用户的自然语言描述" [maxResults=10] [minSimilarity=0.5]
```

查询参数说明：
- `query`：自然语言描述，不要用关键词拼凑。越接近真实表达的句子效果越好。例如：
  - ✅ "关于 MCP 服务器安全配置的讨论"
  - ✅ "之前分析过 embedding 模型的优缺点"
  - ❌ "MCP 安全"（过于碎片化的关键词）
- `maxResults`：返回结果数（1-30，默认 10）
- `minSimilarity`：相似度阈值（0-1，默认 0.5）。阈值越低召回越多但噪音越大
- `format`：输出格式（text/json）

### 5. 结果解读

搜索结果包含：
- **匹配度百分比**：语义相似度评分
- **文件名/日期**：来源会话
- **话题标签**：自动提取的关键话题
- **上下文预览**：对话片段的前几轮，帮助快速判断是否要找的内容

如果第一次搜索不理想，可以：
- 换一种表述重试（不同角度描述同一件事）
- 降低 minSimilarity 展开召回
- 从结果的话题标签中获得灵感，再搜索

## 技术说明

- 嵌入模型：默认使用 Ollama 本地服务的 `nomic-embed-text`（可配置）
- 分片策略：以 N 轮对话为单位滑动窗口切片，保证上下文完整性
- 向量维度：取决于嵌入模型（nomic-embed-text 为 768 维）
- 相似度计算：余弦相似度
- 索引存储：JSON 文件，位于插件数据目录
- 自动增量索引：每 30 分钟扫描一次新会话

## 注意事项

- 嵌入服务首次调用时会自动加载模型到内存，第一次请求可能较慢（几秒到十几秒）
- `nomic-embed-text` 模型约 274MB，首次使用时 Ollama 会自动下载
- 索引量大时（上万片段），搜索延迟可能增加。nomic-embed-text 每次搜索约需 10-50ms 的计算时间
- 如果用户提到"效率太低"或"搜索太慢"，可以考虑升级嵌入模型或调整 chunkSize
