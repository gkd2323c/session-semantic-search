# Session Semantic Search

## 核心价值

过往对话的语义检索。对话的价值不在标题里，而在对话过程之中。用户可能记不清当时用了什么词、什么表述，但能描述"大概是什么事"——这正是语义搜索擅长的。

## 何时使用

当用户提出以下类型的问题时，优先使用 `session-semantic-search_search` 的 `search` 动作：

- "之前我们讨论过 XXX 相关的内容"
- "我记得你说过关于 XXX 的事，帮我找找"
- "我们是不是聊过 XXX 话题"
- "上次那个关于 XXX 的方案/想法/代码"
- "你之前分析过 XXX 的问题对吧"
- 任何涉及查阅过往对话的需求

不要等用户明确说出"搜一下历史"——只要语义上暗示需要翻看历史，就是触发电机。

## 使用流程

### 1. 检查服务状态
```
search action=health
```
确认嵌入服务（Ollama）在线。如果不在线，提醒用户启动 Ollama。

### 2. 检查索引状态
```
search action=stats
```
看索引中有多少会话和片段。如果为空（totalChunks=0），先做一次全量索引。

### 3. 索引管理
- **自动机制**：插件每 30 分钟自动扫描，支持**增量追加**更新，并能自动检测会话的结构性变动（如被手动改写），触发全量自愈。
- **强制重建**：如果索引状态明显异常（如内容明显缺失），使用：
```
search action=reindex
```

### 4. 语义搜索（核心功能）
```
search action=search query="用户的自然语言描述" [maxResults=10] [minSimilarity=0.5]
```

查询参数说明：
- `query`：自然语言描述，不要用关键词拼凑。越接近真实表达的句子效果越好。例如：
  - ✅ "关于 MCP 服务器安全配置的讨论"
  - ✅ "之前分析过 embedding 模型的优缺点"
  - ❌ "MCP 安全"（过于碎片化的关键词）
- `maxResults`：返回结果数（1-30，默认 10）
- `minSimilarity`：相似度阈值（0-1，默认 0.5）。阈值越低召回越多但噪音越大

### 5. 结果解读
搜索结果包含匹配度评分、来源会话、自动提取的话题标签及上下文片段。

## 技术说明

- **智能索引引擎**：
  - **原子问答切分**：采用原子化问答切分策略，确保语义上下文连续、完整，避免切断问答对。
  - **智能增量追加**：针对新增对话进行快速追加索引，显著降低 IO 开销。
  - **结构变动自愈**：能够自动检测会话文件的结构性变动（篡改、编辑、重写），并自动触发全量重建以保障数据一致性。
- **嵌入模型**：默认使用 Ollama 本地服务的 `nomic-embed-text`（可配置）。
- **技术架构**：暴力余弦相似度计算，高性能 JSON 存储，支持高并发轻量化检索。

## 注意事项

- 嵌入服务首次调用时会自动加载模型到内存，第一次请求可能较慢。
- 自动增量索引在后台安静执行，无需用户参与。
- 若用户报告搜索不准确，优先引导用户更换表述方式或调整 `minSimilarity`。
